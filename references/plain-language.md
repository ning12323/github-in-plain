# 大白话写作规则

读者：**完全没写过代码**。可能是长辈、产品经理、创业者，只是好奇「这个链接是啥」。

## 怎么说

- 像跟朋友聊天，不要像写论文
- 句子短，一句一个意思
- 用「你」和「它」
- 能不用英文就不用；产品名（如 FastAPI、Vite）可以保留

## 必须有的 7 个标题（逐字一致）

```markdown
## 三句话讲明白
## 打个比方
## 它能帮你干什么
## 大概怎么运作的（不用懂代码）
## 五个你可能关心的问题
## 用不用管它
## 和类似的东西比一比
```

最后可加 `## 给懂技术的人（可选）`，最多 5 行，**必须放在全文最末尾**。

## 五个固定问题（必须逐个回答）

1. **免费吗？能商用吗？**
2. **还在维护吗？会不会跑路？**
3. **难不难上手？**（分「普通人」和「程序员」两句话说）
4. **有没有类似的？**（列 2～3 个，各一句话）
5. **有什么要注意的？**

## 专业词怎么说（出现就必须解释）

| 词 | 大白话解释 |
|----|-----------|
| 开源 | 代码公开，一般免费，谁都能看 |
| GitHub | 放程序项目的网站，像「程序员的展示柜」 |
| 框架 | 搭软件用的半成品模板，不用从零开始 |
| 接口 / API | 程序之间「递纸条」的方式，像餐厅菜单 |
| 前端 | 你看到的网页、App 界面 |
| 后端 | 界面背后处理数据的部分，像餐厅后厨 |
| 部署 | 把软件放到服务器上，让别人能访问 |
| 星标 Star | GitHub 上的「收藏/点赞」，人多通常表示关注度高 |
| MIT 许可证 | 很宽松的使用规则，一般可免费商用 |

## 正文里不要出现

- 文件路径（如 `src/main.py`）
- 分数评分（如 4/5）
- 英文技术缩写不解释就扔出来（ASGI、CI、ORM…）
- 架构图、mermaid
- Adopt / Evaluate / Avoid 这类术语

## 类比从哪找

至少写 **2 处**类比，用这些套路：

- 像 **餐厅**：前台点菜 = 用户操作，后厨 = 程序处理
- 像 **乐高**：底板已经拼好，你只加自己的积木
- 像 **快递**：下单 → 分拣 → 送达
- 像 **手机 App**：你点的按钮背后有服务器在干活

## README 太简略时

写清楚：「项目说明写得不多，下面主要根据 GitHub 简介和页面信息整理。」不要编造功能。
